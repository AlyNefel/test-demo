Git Flow — Simple Explanations & Analogies

Quick, practical explanations of core Git concepts with everyday analogies. Includes common commands.

1. What is a Git repository?

Analogy: A repository is like a kitchen or a workshop where you keep all your ingredients, tools and recipes.

It stores the entire history of your project, including snapshots of files and the timeline of changes.

2. git init — Start the kitchen

Analogy: git init is like opening a new kitchen or workshop and putting a label on the door: "This is the place we track work."

git init

It creates the hidden .git folder that will track changes.

3. git add — Put items in your basket

Analogy: You are shopping in the grocery store and put selected items into your basket. git add chooses which file changes will go into the next save.

git add file.txt
git add .      # stage all changed files

Staging prepares a set of changes for the next commit.

4. git commit — Take a snapshot

Analogy: Commit is taking a photo of your kitchen state — what you have and how things are arranged — with a caption explaining why you took the photo.

git commit -m "Describe what changed"

Commits are the saved points in history you can return to or compare.

5. Branches — Parallel workspaces

Analogy: Branches are parallel worktables in the same kitchen. Each table lets someone try a different recipe without disturbing the main service.

  • main (or master) = the main menu / production-ready code
  • Feature branches = developer's separate table for a new feature
git branch          # list branches
git checkout -b feat/login   # create & switch to a feature branch

6. git merge / git rebase — Bring changes together

Merging: like taking the best dishes from one table and adding them to the main menu by combining events.

Rebasing: like replaying a sequence of steps from one table on top of the main table so the timeline looks linear.

git checkout main
git merge feat/login
# or
git checkout feat/login
git rebase main

7. Remote & Push/Pull — Share with the restaurant chain

A remote repository (like origin) is the central kitchen in the chain. Pushing sends your changes there; pulling brings others' changes to you.

git remote add origin <url>
git push origin feat/login
git pull origin main

8. Pull Request (PR) / Merge Request — Ask for review

Analogy: A pull request is like asking the head chef to taste and approve your new dish before it's added to the main menu. You present what you changed, why, and request feedback or approval.

  • PR shows diffs, discussion, and automated tests.
  • After approval, the branch is merged into the main branch (the dish goes on the menu).

PRs typically happen on platforms like GitHub, GitLab or Bitbucket — the UI handles the conversation, reviews and merge button.

9. Quick workflow summary

# initialize repository
git init

# create feature branch
git checkout -b feat/your-feature

# work: stage and commit often
git add .
git commit -m "Work in progress"

# push to remote and open a PR for review
git push -u origin feat/your-feature
# open PR on hosting site, get review, then merge

Keep commits small and descriptive. Use branches for isolated work. Use PRs for collaboration and review.
